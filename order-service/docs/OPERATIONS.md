# Order Service Operations Guide

**Version:** 1.0.0  
**Status:** Implemented runbook  
**Internal service port:** `8082`

## Current State

`order-service` includes the Maven build, Java implementation, automated tests,
container image definition, Compose files, and Postman/Newman assets described
by this runbook. Host tests, Javadoc generation, local startup, and local Newman
verification are executable from this directory. Gateway JWT end-to-end checks
still require a configured running `gateway-service` and suitable test tokens.

## Prerequisites

- Java 25
- Maven wrapper committed by the service
- Podman with Compose support
- Node.js/npm for Newman
- running `gateway-service` for full JWT end-to-end tests
- shared internal network `sentra-gateway_services`

## Podman Base Deployment

Create the shared internal network only when the gateway stack has not already
created it:

```powershell
podman network create --internal sentra-gateway_services
```

Build and run without publishing the application port:

```powershell
Copy-Item .env.example .env
podman compose --env-file .env config
podman compose --env-file .env up --build -d
podman compose ps
```

The base `compose.yaml` must:

- use `expose: 8082`, not a host `ports` mapping;
- attach to `sentra-gateway_services`;
- run non-root;
- use a read-only root filesystem;
- set `no-new-privileges`;
- mount bounded `/tmp` as `tmpfs`;
- define a readiness health check.

## Local Postman Deployment

For direct API testing, combine the base file with a local-only override:

```powershell
podman compose --env-file .env `
  -f compose.yaml `
  -f compose.postman.yaml `
  up --build -d
```

The override must publish:

```text
127.0.0.1:8082 -> container 8082
```

With Podman, a container attached only to internal networks cannot receive host
port forwarding. The Postman override therefore also needs a non-internal local
test network. This network must not appear in the base deployment.

## Direct Smoke Tests

These calls simulate headers generated by the gateway and are allowed only for
local verification.

Common headers:

```powershell
$readHeaders = @{
  "X-Sentra-Request-Id" = "local-orders-read-001"
  "X-Sentra-Subject" = "sentra-user-omar"
  "X-Sentra-Actor-Type" = "USER"
  "X-Sentra-Tenant-Id" = "tenant-demo"
  "X-Sentra-Scopes" = "orders:read"
  "X-Sentra-Route-Id" = "orders-list"
}
```

List owned orders:

```powershell
Invoke-RestMethod `
  -Uri "http://localhost:8082/internal/v1/orders?page=0&size=20" `
  -Headers $readHeaders
```

Get an owned order:

```powershell
$getHeaders = $readHeaders.Clone()
$getHeaders["X-Sentra-Route-Id"] = "orders-get"

Invoke-RestMethod `
  -Uri "http://localhost:8082/internal/v1/orders/10000000-0000-4000-8000-000000000001" `
  -Headers $getHeaders
```

Create an order:

```powershell
$writeHeaders = @{
  "X-Sentra-Request-Id" = "local-orders-create-001"
  "X-Sentra-Subject" = "sentra-user-omar"
  "X-Sentra-Actor-Type" = "USER"
  "X-Sentra-Tenant-Id" = "tenant-demo"
  "X-Sentra-Scopes" = "orders:write"
  "X-Sentra-Route-Id" = "orders-create"
  "Idempotency-Key" = "postman-order-create-001"
}

$body = @{
  items = @(
    @{ sku = "BOOK-JAVA-25"; quantity = 1 }
  )
} | ConvertTo-Json -Depth 5

Invoke-WebRequest `
  -Method Post `
  -Uri "http://localhost:8082/internal/v1/orders" `
  -Headers $writeHeaders `
  -ContentType "application/json" `
  -Body $body
```

Repeat the exact request with a different request ID and the same idempotency key.
Expected result:

- `201`;
- same order body and `Location`;
- `Idempotency-Replayed: true`;
- one stored order.

Reuse the key with a different item request. Expected result:

```text
409 ORD_IDEMPOTENCY_CONFLICT
```

Administrator list:

```powershell
$adminHeaders = @{
  "X-Sentra-Request-Id" = "local-admin-orders-001"
  "X-Sentra-Subject" = "sentra-admin"
  "X-Sentra-Actor-Type" = "USER"
  "X-Sentra-Roles" = "ORDER_ADMIN"
  "X-Sentra-Route-Id" = "admin-orders-list"
}

Invoke-RestMethod `
  -Uri "http://localhost:8082/internal/v1/admin/orders?page=0&size=20" `
  -Headers $adminHeaders
```

## Postman And Newman

Required collection assets:

```text
postman/Sentra_Order_Service.postman_collection.json
postman/Sentra_Order_Service_Local.postman_environment.json
```

Required collection flow:

1. readiness is `UP`;
2. OpenAPI contains all four operations;
3. owned list returns only the current subject and tenant partition;
4. owned single read succeeds;
5. foreign subject returns `404`;
6. foreign tenant returns `404`;
7. missing trusted context returns `401`;
8. wrong actor, scope, role, or route returns `403`;
9. create returns `201`;
10. same key/same payload replays the original order;
11. same key/different payload returns `409`;
12. malformed, unknown, duplicate-SKU, and oversized bodies are rejected;
13. admin list includes owner references;
14. restart restores deterministic local state.

CLI:

```powershell
npx --yes newman run `
  postman/Sentra_Order_Service.postman_collection.json `
  -e postman/Sentra_Order_Service_Local.postman_environment.json `
  --reporters cli
```

## Gateway End-To-End Verification

Direct trusted-header tests do not prove JWT behavior. A release candidate must
also call the external gateway routes:

```text
GET  /api/v1/orders
GET  /api/v1/orders/{id}
POST /api/v1/orders
GET  /api/v1/admin/orders
```

Required JWT scenarios:

- valid user with `orders:read`;
- valid user without `orders:read`;
- valid user with `orders:write`;
- expired/invalid JWT;
- valid non-admin user on admin route;
- valid user carrying `ORDER_ADMIN`;
- external spoofed `X-Sentra-Subject` cannot change ownership;
- bearer token is not forwarded to `order-service`.

## Monitoring

Verify required metrics:

```powershell
$metrics = Invoke-WebRequest `
  -UseBasicParsing `
  http://localhost:8082/actuator/prometheus

$metrics.Content | Select-String "sentra_order_"
```

Inspection must confirm that labels never include subjects, tenants, order IDs,
request IDs, SKUs, raw paths, or idempotency keys.

Suggested alerts:

- readiness unavailable;
- elevated `5xx` ratio;
- p95 order latency above approved threshold;
- repository failures;
- idempotency capacity rejection;
- unusual permission-denial spike;
- Prometheus scrape failure.

## Failure Behavior

| Failure | Required behavior |
| --- | --- |
| Missing gateway provenance | `401 ORD_TRUSTED_CONTEXT_REQUIRED` |
| Missing scope/role | stable `403` |
| Foreign order | indistinguishable `404` |
| Invalid create body | stable `400` |
| Oversized body | `413` before domain work |
| Idempotency payload conflict | `409`, no new order |
| Idempotency capacity exhausted | `503`, no unsafe live-record eviction |
| Repository unavailable | `503`, readiness down where unsafe |
| Create timeout at gateway | no automatic retry without preserved idempotency contract |
| Metrics unavailable | service traffic continues; alert |

The service must never silently weaken ownership or trusted-context enforcement.

## Incident Checks

### Unexpected Foreign Data

1. Stop direct traffic and preserve logs.
2. Correlate by request ID and route ID.
3. Verify subject and tenant filtering happened in the repository/service layer.
4. Check duplicate/malformed trusted headers and gateway peer validation.
5. Confirm no user DTO exposes owner fields.
6. Treat confirmed cross-owner disclosure as a security incident.

### Duplicate Orders

1. Identify whether `Idempotency-Key` was present.
2. Verify gateway retries were disabled or preserved the same key/body.
3. Check atomic uniqueness for the idempotency scope.
4. Check retention/capacity and cleanup behavior.
5. Do not delete evidence before recording order and request references securely.

### Elevated Denials

1. Split by finite result code and route ID.
2. Compare gateway scope/role policy with service route policy.
3. Verify trusted list encoding compatibility.
4. Check whether route IDs changed during deployment.
5. Avoid enabling bypass configuration as a recovery action.

## Runtime Data

Memory mode is non-durable:

- restart restores deterministic seed data;
- created orders and idempotency records are lost;
- horizontal replicas do not share writes;
- backup and restore are not applicable.

Production durability requires an approved external repository design before
deployment.
